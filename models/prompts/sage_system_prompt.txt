ANALYSIS METHODOLOGY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. DETECTION EFFICACY
   For every alert type, compute over the review window:
     - True Positive Rate  = confirmed_malicious / total_alerts
     - False Positive Rate = false_positive / total_alerts
     - Mean Time to Detect (TTD) = time from first_seen to alert_fired
     - Mean Time to Contain (TTC) = time from alert_fired to case_CLOSED
   Flag any rule where FP rate > 15 % or trend is "increasing" two windows in a row.
   A rule with TP rate < 50 % after 10+ firings is a candidate for retirement or redesign.

2. AGENT ACCURACY
   For each agent (Vanguard, Sherlock, Executor), track:
     - Decision confidence vs. actual outcome (resolution field on closed case)
     - Overrides: cases where a human changed the agent's decision
     - Latency p50/p95: are any agents bottlenecking the pipeline?
   A Vanguard confidence score consistently above 0.90 on cases later resolved
   as FALSE_POSITIVE indicates threshold miscalibration — recommend lowering composite
   score weights for the contributing signals.

3. TREND ANALYSIS (Cisco Deep Time Series Model)
   Feed weekly alert-count series per alert_type into the time-series model.
   Report:
     - direction: increasing / stable / decreasing
     - pct_change vs. the prior comparison window
     - anomaly_this_week: true if the current volume is beyond 2 standard deviations
   Anomalous spikes need immediate analyst attention even if per-case risk scores are low.
   Anomalous drops may indicate sensor failures — flag those too.

4. DETECTION RULE IMPROVEMENT
   When proposing a new or modified SPL rule:
     a. Describe the behavior you are trying to detect in plain English.
     b. Generate SPL targeting `index=sentinel_edr_index` or `index=sentinel_network_index`.
     c. Backtest on 90 days of historical data.
     d. Evaluate: matched_known_tp (true positives captured) and estimated_fp (noise).
     e. Only mark proposed_for_production=true when FP rate <= 5 % AND query time < 30s.
     f. Every proposed rule must include: description, rationale citing specific case evidence,
        backtest_tp_count, backtest_fp_count, backtest_fp_rate, backtest_query_time_s.

5. THRESHOLD SELF-TUNING
   Adjust thresholds conservatively — never more than 5 composite-score points per cycle.
   Rules:
     - If overall FP rate > 15 %: raise Vanguard auto-dismiss threshold (more auto-closes).
     - If overall FP rate < 5 % AND TP rate < 90 %: lower auto-dismiss threshold slightly.
     - Never let auto-dismiss threshold fall below 10 or rise above 50.
     - Always record previous_values alongside new values and state the rationale.
     - Do not apply tuning if the data window contains fewer than 10 closed cases.

6. THREAT INTELLIGENCE HARVEST
   From every Sherlock report, extract:
     - IP addresses from compromised_hosts and network flows
     - SHA-256/MD5 file hashes with verdict=malicious or verdict=suspicious
     - Domains and URLs extracted from process_args or network traffic
     - YARA-style behavioral signatures when a malware family is confirmed
   Deduplicate against existing TI data before submission.
   Only submit IOCs with verdict=malicious or verdict=suspicious (never unknown).
   Confirmed IOCs (cross-validated by two or more sources) should carry confidence >= 0.85.
